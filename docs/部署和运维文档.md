## 1. 部署概览

Echo Tours 通过 Docker 容器化实现一键式部署，支持以下平台：

| 平台 | 方式 | 说明 |
|------|------|------|
| **Mac 电脑** | Docker Compose | 开发/演示/本地生产 |
| **Linux 服务器** | Docker Compose | 自托管生产环境 |
| **AWS EC2** | Docker Compose + 远程同步 | 通过 SSH 推送并部署 |
| **AWS ECS** | ECR + ECS 服务 | 云原生容器编排 |
| **GCP Cloud Run** | Artifact Registry + Cloud Run | 无服务器容器部署 |

## 2. 环境要求

### 硬件要求

| 资源 | 最低要求 | 推荐配置 |
|------|---------|---------|
| CPU | 2 核 | 4 核 |
| 内存 | 4 GB | 8 GB |
| 磁盘 | 10 GB | 20 GB |

**内存分布说明（含所有服务）：**

| 服务 | 占用 |
|------|------|
| Elasticsearch | ~1 GB |
| PostgreSQL | ~256 MB |
| Redis | ~64 MB |
| Backend (FastAPI) | ~256 MB |
| Frontend (Next.js) | ~256 MB |
| Nginx | ~32 MB |
| Celery Worker | ~128 MB |
| **总计** | **~2 GB** |

> ⚠️ Elasticsearch 是内存消耗大户。若在资源有限的机器上部署，可注释掉 `docker-compose.prod.yml` 中的 elasticsearch 服务，应用会自动降级运行。

### 软件要求

- **Docker** >= 24.0（[下载地址](https://docs.docker.com/get-docker/)）
- **Docker Compose** v2（Docker Desktop 已内置）
- **Make**（macOS 和大多数 Linux 已预装）

## 3. 一键部署（本地/第三方服务器）

### 3.1 首次完整部署（推荐）

```bash
# 1. 克隆仓库
git clone <仓库地址>
cd Echo-Website

# 2. 一键部署
#    脚本会自动完成：环境检查 → 生成 .env → 拉取基础镜像 → 构建应用镜像 → 启动服务 → 执行迁移
make setup

# 或者直接执行脚本
bash scripts/setup.sh
```

**部署成功后的输出：**

```
╔══════════════════════════════════════╗
║   🎉 Echo Tours 部署完成！           ║
╠══════════════════════════════════════╣
║   Frontend:  http://localhost        ║
║   API:       http://localhost/api/v1 ║
║   Docs:      http://localhost/docs   ║
╚══════════════════════════════════════╝
```

### 3.2 快速启动（已有镜像）

```bash
make prod
```

### 3.3 带演示数据的部署

```bash
bash scripts/setup.sh --with-seed
```

### 3.4 仅构建镜像（不启动）

```bash
make build
# 或
bash scripts/setup.sh --skip-migrate
```

## 5. 服务架构

### 5.1 架构图

```
用户请求
    │
    ▼
┌──────────┐   Nginx (:80)   统一入口
│  Nginx   │   ├── /api/*       → backend:8000 （FastAPI + Uvicorn）
│  Alpine  │   ├── /docs        → backend:8000 （Swagger 文档）
│          │   ├── /health      → backend:8000 （健康检查）
│          │   └── /*           → frontend:3000（Next.js）
└──────────┘
     │
     ├──────────────────────────────┐
     ▼                              ▼
┌──────────┐                 ┌──────────┐
│ backend  │                 │ frontend │
│ FastAPI  │◄───────────────►│ Next.js  │
│ 4workers │    Docker 网络   │ standalone│
└────┬─────┘                 └──────────┘
     │
     ├──────────┬──────────┬──────────┐
     ▼          ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│Postgres│ │ Redis  │ │Elastic │ │ Celery │
│  16    │ │   7    │ │Search 8│ │ Worker │
└────────┘ └────────┘ └────────┘ └────────┘
```

### 5.2 服务列表

| 服务名 | 镜像 | 内部端口 | 说明 |
|--------|------|---------|------|
| `nginx` | nginx:alpine | 80 | 反向代理网关 |
| `postgres` | postgres:16-alpine | 5432 | 主数据库 |
| `redis` | redis:7-alpine | 6379 | 缓存 / Celery 消息代理 |
| `elasticsearch` | elasticsearch:8.17.0 | 9200 | 全文搜索引擎（可选） |
| `migrations` | 自定义 (prod) | - | 数据库迁移（一次性运行后退出） |
| `backend` | 自定义 (prod) | 8000 | FastAPI 应用服务 |
| `frontend` | 自定义 (prod) | 3000 | Next.js 前端 |
| `celery_worker` | 自定义 (prod) | - | 异步任务消费者 |
| `celery_beat` | 自定义 (prod) | - | 定时任务调度器 |

## 6. 环境变量配置

### 6.1 自动生成

首次执行 `setup.sh` 时，脚本会：

1. 从 `.env.example` 复制创建 `.env` 文件
2. 使用 `openssl rand -hex 32` 生成随机 JWT 密钥
3. 替换 `.env` 中的默认密钥占位符

### 6.2 必须配置的变量

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `POSTGRES_USER` | 数据库用户名 | `postgres` |
| `POSTGRES_PASSWORD` | 数据库密码 | `postgres` |
| `SECRET_KEY` | JWT 密钥（自动生成） | 32 字节随机 hex |

### 6.3 可选配置（按需设置）

| 变量 | 说明 | 不设置的影响 |
|------|------|-------------|
| `STRIPE_SECRET_KEY` | Stripe 支付密钥 | 支付功能不可用 |
| `STRIPE_PUBLIC_KEY` | Stripe 公钥 | 前端支付按钮不可用 |
| `GOOGLE_CLIENT_ID` | Google OAuth 客户端 ID | Google 登录不可用 |
| `GOOGLE_CLIENT_SECRET` | Google OAuth 密钥 | Google 登录不可用 |
| `SENDGRID_API_KEY` | 邮件发送密钥 | 邮件通知不可用 |
| `CORS_ORIGINS` | 允许的跨域域名 | 仅接受默认 localhost |

### 6.4 修改配置后重启

```bash
# 修改 .env 文件后需要重建并重启服务
make down
make prod
```

## 7. 日常运维命令

### 7.1 服务管理

```bash
make up        # 启动生产服务
make down      # 停止所有服务
make restart   # 重启生产服务
make logs      # 查看实时日志（Ctrl+C 退出）
make dev       # 启动开发环境（热重载）
```

### 7.2 数据库

```bash
make psql      # 进入 PostgreSQL 交互终端
make seed      # 填充演示数据（开发/演示用）
```

### 7.3 其他

```bash
make build     # 重建生产镜像
make test      # 运行后端测试
make clean     # 清理所有容器、卷、镜像（数据将丢失）
make help      # 显示所有可用命令
```

### 7.4 手动执行迁移

```bash
# 如果自动迁移失败，可手动执行
docker compose -f docker-compose.prod.yml run --rm migrations

# 或直接执行
docker compose -f docker-compose.prod.yml exec backend alembic upgrade head
```

### 7.5 查看服务状态

```bash
docker compose -f docker-compose.prod.yml ps
docker compose -f docker-compose.prod.yml top
```

### 7.6 查看单个服务日志

```bash
docker compose -f docker-compose.prod.yml logs backend
docker compose -f docker-compose.prod.yml logs -f --tail 100 frontend
```

## 8. Docker 镜像构建

### 8.1 构建流程

```
后端（Python 3.12-slim）:
  源码 → pip install 依赖 → 复制源码 → HEALTHCHECK → 启动 Uvicorn (4 workers)

前端（Node 20-alpine）:
  源码 → npm install → next build → 复制 standalone 输出 → 启动 server.js
```

### 8.2 多阶段构建

**后端 Dockerfile 目标：**

| 目标 | 用途 | 说明 |
|------|------|------|
| `dev` | 开发环境 | 包含热重载和开发依赖 |
| `prod` | 生产环境 | 压缩安装、非 root 用户、健康检查 |

**前端 Dockerfile 目标：**

| 目标 | 用途 | 说明 |
|------|------|------|
| `dev` | 开发环境 | 源码挂载，热重载 |
| `builder` | 构建阶段 | 编译 TypeScript，生成 standalone 输出 |
| `prod` | 生产环境 | 仅复制运行时必须的文件 |

### 8.3 构建优化（.dockerignore）

`.dockerignore` 文件确保构建上下文只包含必要文件，提升构建速度并减小镜像体积：

- **后端排除**：`tests/`、`__pycache__`、`.env`、`*.egg-info`
- **前端排除**：`node_modules/`、`.next/`、`.env`

## 9. AWS 部署

### 9.1 AWS EC2 部署

将项目通过 SSH 同步到 EC2 实例并直接使用 Docker Compose 运行：

```bash
# 部署到远程 EC2
bash scripts/deploy.sh --cloud ec2 --ssh-user ec2-user --ssh-host <EC2-PUBLIC-IP>

# 后访问 http://<EC2-PUBLIC-IP>
```

**前置条件：**
- EC2 实例安全组开放端口 80（HTTP）
- EC2 实例已安装 Docker（脚本会自动安装）
- 本地有 SSH 私钥访问权限

### 9.2 AWS ECS 部署

使用 ECR + ECS Fargate 云原生部署：

```bash
# 设置环境变量
export AWS_ACCOUNT=123456789012
export AWS_REGION=us-east-1

# 构建并推送镜像到 ECR，然后更新 ECS 服务
bash scripts/deploy.sh --cloud aws --env prod
```

**前置条件：**
- AWS CLI 已安装并配置
- AWS CLI 有足够的 ECR/ECS 权限
- ECS 集群和任务定义已准备好

### 9.3 AWS 部署注意事项

- ECS 部署时需要提前创建好 ECR 仓库和 ECS 集群
- 对于 RDS（数据库）、ElastiCache（Redis）、OpenSearch 等服务，建议使用托管服务而非 Docker 容器
- 将 `docker-compose.prod.yml` 中的数据库连接指向 AWS 托管服务

## 10. GCP 部署

## 11. 常见问题排查

### 11.1 端口被占用

```bash
# 检查端口占用
lsof -i :80 -i :3000 -i :8000

# 修改 docker-compose.prod.yml 中映射的端口
# 将 "80:80" 改为 "8080:80" 等
# 或通过 .env 设置:
# NGINX_PORT=8080
```

### 11.2 磁盘空间不足

```bash
# 清理未使用的 Docker 资源
docker system prune -a --volumes
# ⚠️ 这将删除所有未使用的容器、镜像和卷
```

### 11.3 Elasticsearch 内存不足

```yaml
# 编辑 docker-compose.prod.yml，减少 ES 内存
environment:
  - "ES_JAVA_OPTS=-Xms256m -Xmx256m"  # 从 512m 改为 256m
deploy:
  resources:
    limits:
      memory: 512M  # 从 1G 改为 512M
```

### 11.4 数据库迁移失败

```bash
# 查看迁移日志
docker compose -f docker-compose.prod.yml logs migrations

# 手动重置并重新迁移
docker compose -f docker-compose.prod.yml exec postgres psql -U postgres -d echo_tours -c "DROP TABLE IF EXISTS alembic_version;"
docker compose -f docker-compose.prod.yml run --rm migrations
```

### 11.5 前端空白页面

```bash
# 1. 检查前端日志
docker compose -f docker-compose.prod.yml logs frontend

# 2. 检查 API 是否可达
curl http://localhost/api/v1/tours

# 3. 检查 Nginx 日志
docker compose -f docker-compose.prod.yml logs nginx
```

### 11.6 Docker Desktop Mac 资源设置

Docker Desktop 默认内存可能不足（尤其 ES 需要 1G+）：

1. 打开 Docker Desktop
2. Settings → Resources → Advanced
3. 设置 Memory 为至少 6 GB
4. Apply & Restart

## 12. CI/CD 工作流

### 12.1 CI 工作流（`.github/workflows/ci.yml`）

在每次 push 到 `main` 和 `develop` 分支时自动运行：

| 步骤 | 内容 |
|------|------|
| `lint-backend` | Ruff 代码检查 |
| `lint-frontend` | ESLint + TypeScript 类型检查 |
| `test-backend` | Pytest（含 PostgreSQL + Redis 服务容器） |
| `build-docker` | 验证 Docker Compose 构建 |

### 12.2 CD 工作流（`.github/workflows/deploy.yml`）

在 push 到 `main` 分支时自动构建并推送容器镜像：

- 推送到 GitHub Container Registry（ghcr.io）
- 同时生成 `latest` 和 commit SHA 标签
- 也支持手动触发（`workflow_dispatch`），可选择环境（staging/production）和目标平台

### 12.3 手动触发 CD

在 GitHub 仓库页面：

1. 点击 **Actions** 标签
2. 选择 **Deploy** 工作流
3. 点击 **Run workflow**
4. 选择环境和目标平台
5. 点击 **Run**
